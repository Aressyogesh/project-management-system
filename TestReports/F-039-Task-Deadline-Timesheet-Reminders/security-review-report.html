OWASP Top 10 Security Checks
IDCheckStatusNotes
SEC-1 Broken Access Control N/A No HTTP endpoints; cron jobs are internal server-side processes. No access control surface.
SEC-2 Cryptographic Failures PASS No sensitive data stored. Email addresses passed to SMTP — transport encryption handled by EmailService (SSL/TLS on port 465). No credentials in cron service.
SEC-3 Injection (SQL/NoSQL/Command) PASS All DB queries use Prisma parameterised API (findMany, aggregate). No raw SQL, no string interpolation into queries. Zero injection risk.
SEC-4 Insecure Design PASS Cron jobs query only their required data (assignee email, task title, project name, hours). Principle of least privilege respected in select/include clauses.
SEC-5 Security Misconfiguration PASS No new configuration surface introduced. Cron schedules are hardcoded constants — cannot be changed via env/API at runtime.
SEC-6 Vulnerable & Outdated Components PASS @nestjs/schedule is an official NestJS package. No new third-party dependencies introduced by this feature.
SEC-7 Identification & Authentication Failures N/A Cron jobs run server-side; no authentication required. Email recipient addresses come from the internal user database — no user-supplied input.
SEC-8 Software & Data Integrity Failures PASS Email body assembled from DB-sourced strings (task title, project name, user fullName). No deserialization of external data. No supply-chain risk from new deps.
SEC-9 Security Logging & Monitoring Failures PASS Each cron run logs start/end. SMTP failures logged via logger.error with recipient email and error message. NestJS Logger integrates with PM2 log rotation.
SEC-10 Server-Side Request Forgery (SSRF) N/A No outbound HTTP requests. SMTP sending is via EmailService which uses a fixed configured server — not influenced by any user input.
SEC-11 Email Content Injection / Header Injection PASS Email subject constructed from safe integer (task count) and static strings. Body content (task title, project name, user name) comes from the internal database. No user-supplied strings reach the email subject or headers at this layer.
Additional Infrastructure Checks
CheckStatusNotes
PII in logs PASS Only email address logged on SMTP failure (necessary for debugging). No passwords, tokens, or sensitive data logged.
Error message leakage PASS Errors caught and logged server-side only; never propagated to any HTTP response.
Denial of Service via cron flood PASS Named cron jobs prevent duplicate scheduling. @nestjs/schedule ensures each job runs once per interval even under load. No user-triggerable endpoint exists for these jobs.
SMTP credential security PASS SMTP credentials stored in env vars (GitHub Actions secrets → PM2 env). Not hardcoded in service or config files.
