Security Review Report — F-017: Task Allocation CRUD + 8h Daily Cap Validation

Reviewer: Claude Sonnet 4.6  |  Date: 2026-05-27

Verdict: CLEARED — No critical or high findings.

OWASP Checks

CheckFindingStatus
SEC-1 (A01) Broken Access ControlAll write endpoints (POST, PATCH, DELETE) protected by JwtAuthGuard + ProjectRoleGuard. Only PM/TL project roles can create/update/delete allocations. EMPLOYEE cannot perform write operations — returns 403. Read endpoints (GET check, GET by user, GET by project) require JWT but do not enforce project-role restriction, which is appropriate for visibility.PASS
SEC-1 (A01) Object-Level AuthPATCH/DELETE use 'allocation' source in ProjectRoleGuard — resolves projectId from the allocation's task chain (allocation → task → projectId) in the database. An attacker cannot manipulate a different project's allocation by guessing an ID; the guard verifies project membership before any mutation.PASS
SEC-3 (A03) InjectionAll query parameters (userId, date, taskId) used as Prisma where clause values — ORM parameterised queries prevent SQL injection. No raw SQL used. DTO validators (@IsUUID, @IsDateString, @IsNumber) enforce input types before reaching the service layer.PASS
SEC-4 (A04) Insecure DesignDaily cap (8h) enforcement is server-side in assertCap() — not relying on frontend validation alone. Frontend cap warning is UX-only; backend rejects over-allocation regardless of client behaviour.PASS
SEC-5 (A05) MisconfigurationNew module (TaskAllocationsModule) registered in AppModule with no additional global configuration changes. CORS, rate limiting, and environment settings unchanged.PASS
API-01 Object Level AuthCovered above — guard resolves ownership via DB lookup, not client-provided projectId on PATCH/DELETE.PASS
API-03 Property Level AuthAllocation response exposes task title, user name, date, hours — no passwords, tokens, or sensitive PII. Included user select is limited to id/name/avatarUrl.PASS
API-06 Mass AssignmentDTOs (CreateTaskAllocationDto, UpdateTaskAllocationDto) explicitly whitelist accepted fields. Prisma create/update only receive mapped DTO fields — no pass-through of raw request body.PASS
