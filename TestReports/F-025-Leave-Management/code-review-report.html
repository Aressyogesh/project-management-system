  • Frontend LeavePage is a single file keeping all leave-specific logic co-located; sub-components (ApplyLeaveModal, RejectModal) are local to the file reducing prop-drilling.

CR-3: React / React Query Best Practices

PASS
  • All useMutation hooks placed unconditionally before JSX returns — hooks ordering rule respected.
  • Leave list query uses queryKey: ['leave-requests', { status, selectedUser }] — cache-busts correctly when filters change.
  • Approval/rejection/cancel mutations each call queryClient.invalidateQueries(['leave-requests']) on success — stale data is evicted immediately.
  • Capacity report query uses staleTime: 60_000 — prevents unnecessary re-fetches on every re-render while keeping data fresh.
  • Toast state is auto-cleared via setTimeout from within the mutation success handler — no leaked timers on unmount for normal flows.

CR-4: API Contract Compliance

PASS
  • All six REST endpoints match the contract defined in REQ-F-025: POST /leave-requests, GET /leave-requests, GET /leave-requests/:id, PATCH …/approve, PATCH …/reject, PATCH …/cancel.
  • Frontend leaveApi.ts calls exactly those endpoints with matching payloads.
  • totalDays is always computed server-side; the frontend never sends it — avoids client-side tampering.
  • Approve and reject accept an optional approvalNote — matches DTO.
  • List endpoint accepts optional status and userId query params; frontend passes them as Axios params object.

CR-5: Database Schema Design

PASS
  • LeaveRequest model uses @db.Date for startDate/endDate — stores date-only value, no spurious time component from UTC conversion.
  • Indexes defined on userId, status, and startDate, endDate — optimal for the overlap check query and admin list queries.
  • Cascading onDelete: Cascade from User — leave records are purged when a user is deleted, no orphaned rows.
  • Two separate named relations (UserLeaveRequests, LeaveRequestApprovals) correctly handle dual FK on User — avoids Prisma ambiguous relation error.
  • npx prisma db push applied successfully to local DB.

CR-7: RBAC & Permission Guards

PASS
  • Service findAll(): SUPER_USER/ADMIN see all records; EMPLOYEE sees only own (where: { userId }).
  • Service approve()/reject(): throws ForbiddenException if caller's role is not SUPER_USER or ADMIN.
  • Service cancel(): EMPLOYEE can only cancel own PENDING leave; ADMIN can cancel any.
  • Frontend actions (Approve, Reject buttons) are conditionally rendered based on canManage (SUPER_USER/ADMIN) — defence-in-depth.
  • Employee column in admin list view is hidden for non-admin users — reduces information exposure.
  • Projects list RBAC: non-admin users now see only projects where they are members (members: { some: { userId } }) in both findAll and getSummary.
