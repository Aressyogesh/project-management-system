SEC-1Broken Access ControlProjectRoleGuard enforces project membership and minimum role for every mutating endpoint. SUPER_USER/ADMIN bypass is intentional and explicit. Non-members receive HTTP 403 — no data leakage path.PASS
SEC-2Cryptographic FailuresNo cryptographic operations introduced. Role checks operate on plain DB-stored enum values. No sensitive data involved.N/A
SEC-3InjectionNo new user-controlled inputs introduced. Guard reads projectId from route param or DB lookup — both paths use Prisma parameterised queries, immune to SQL injection.PASS
SEC-4Insecure DesignRBAC matrix is defined server-side; the frontend hides UI elements as a UX aid only, not a security boundary. Backend always re-validates role on every request regardless of frontend state.PASS
SEC-5Security MisconfigurationJwtAuthGuard remains global — all endpoints require a valid JWT first, then ProjectRoleGuard adds the second check layer. No endpoint is accidentally left unguarded.PASS
SEC-6Vulnerable ComponentsNo new npm packages added. Guard implementation uses existing NestJS/Prisma infrastructure.PASS
SEC-7Auth FailuresGuard short-circuits with 401 if JWT is invalid (handled by JwtAuthGuard upstream). Invalid or expired tokens never reach ProjectRoleGuard. No role bypass via malformed JWT.PASS
SEC-8Software IntegrityNo new external scripts or CDN dependencies. All changes are pure NestJS decorator additions.PASS
SEC-9Logging / Monitoring403 responses from ProjectRoleGuard are captured by NestJS default logging. Unauthorized access attempts are traceable via request logs.PASS
SEC-10SSRFNo outbound HTTP calls introduced. Role resolution is entirely DB-bound.N/A
