OWASP Security Checks (SEC-1 through SEC-11)

CheckOWASP CategoryRiskFindingsResult
SEC-1 A01 Broken Access Control HIGH Auth: JwtAuthGuard is global — all /api/v1/ai/* endpoints require a valid JWT. Verified: unauthenticated requests return 401 (TC-F030-008).
RBAC: User role from JWT is injected into each tool via ctx.systemRole — never trusted from the request body. Employee role enforces assigneeId filter. Admin/Super User have full scope. Verified in unit tests B-004, B-005. 		✓ PASS
SEC-2 A02 Cryptographic Failures LOW JWT signing handled by existing JwtModule (RS256/HS256 per project config) — no new cryptographic code introduced in F-030. conversationId generated with crypto.randomUUID() — cryptographically random, not guessable. AI model key (AI_API_KEY) stored in .env — not committed to git (confirmed in .gitignore). ✓ PASS
SEC-3 A03 Injection (SQL / Prompt) HIGH SQL Injection: All database queries use Prisma ORM with parameterised inputs — zero raw SQL. Tool arguments go through Prisma's type-safe query builder. No injection possible.
Prompt Injection: User message is appended to the Ollama conversation as a user role message only — it cannot override the system prompt or tool definitions. Tool result content is treated as opaque JSON by Ollama and never re-executed. 		✓ PASS
SEC-4 A04 Insecure Design LOW Tools are read-only Prisma queries — no write operations are exposed to AI tool calling. A malicious or confused model cannot mutate data. System prompt explicitly instructs the model to only base answers on tool results — limits hallucination risk. Conversation history limited to last 10 turns to prevent context-window abuse. ✓ PASS
SEC-5 A05 Security Misconfiguration LOW AI_BASE_URL and AI_MODEL loaded from ConfigService — not hardcoded. Swagger docs (@ApiTags) are gated behind dev environment in existing config. No CORS changes made — existing project CORS policy applies. Health endpoint (GET /api/v1/ai/health) requires JWT — no unauthenticated leakage of infra details. ✓ PASS
SEC-6 A06 Vulnerable Components LOW ollama@^0.5.18 is the only new npm dependency. It is a first-party Ollama client library. No known CVEs at time of review. No other new backend or frontend packages introduced in this feature. ✓ PASS
SEC-7 A07 Identification & Authentication Failures HIGH JWT validation is handled by the existing JwtAuthGuard — no custom auth code written in F-030. CurrentUser decorator extracts userId and systemRole from the verified token payload — not from the request body. No session management, no remember-me functionality, no custom token logic added. ✓ PASS
SEC-8 A08 Software & Data Integrity LOW Tool results flow from Prisma → NestJS → Ollama (second call) → client. No external URLs are followed. Ollama host is fixed at AI_BASE_URL env var — no SSRF vector via user input. No file upload or deserialization in this feature. ✓ PASS
SEC-9 A09 Logging & Monitoring LOW Logger (NestJS) used in AiService — tool failures and AI errors are logged at WARN/ERROR level. User message content is not logged (privacy-preserving). Health endpoint returns model reachability without exposing internal config beyond model name. Note: No structured audit log for AI queries — recommended for production but out of scope for Phase 1 local build. ⚠ LOW NOTE
SEC-10 A10 Server-Side Request Forgery (SSRF) HIGH The only outbound HTTP call made by the backend is to AI_BASE_URL — a fixed env var, not user-controlled. No endpoint accepts a URL from the client. No SSRF risk. ✓ PASS
SEC-11 AI-Specific: Data Exfiltration via Model HIGH Model is local (Ollama, llama3.2:3b): No data leaves the local machine to an external AI provider in Phase 1. Zero exfiltration risk from the model layer. Tool scope: Each tool queries only what the RBAC filter permits — an Employee cannot trigger a query that returns another user's data even if the model tries to call the tool with a different userId, because the tool ignores the AI-supplied userId and uses the JWT-extracted one from ctx.userId. ✓ PASS
