#OWASP CategoryFindingStatus
SEC-1Broken Access ControlEMPLOYEE callers have userId forcibly set to their own ID — they cannot query other users' logs by passing a different userId. Admin/SuperUser can query all logs. Audit log table itself has no public write endpoint.PASS
SEC-2Cryptographic FailuresAudit logs contain no sensitive data (no passwords, no tokens). entityTitle stores display-safe strings (name, displayId). Metadata stores action context (status names, role names).PASS
SEC-3InjectionAll query parameters passed through Prisma parameterised queries. Date strings parsed with `new Date()` — invalid dates produce NaN which Prisma rejects safely. No raw SQL in the audit log service.PASS
SEC-4Insecure DesignAudit logs are append-only — no DELETE or UPDATE endpoint exists for audit_logs. The only write path is via internal service call (not exposed as REST). Cascade delete on userId preserves referential integrity when a user is removed.PASS
SEC-5Security MisconfigurationAuditLogsController is protected by the global JwtAuthGuard (no @Public() decorator). All audit log reads require a valid JWT token.PASS
SEC-6Vulnerable ComponentsNo new npm packages added. Uses existing Prisma, NestJS, and React Query infrastructure.PASS
SEC-7Auth FailuresLogin event is logged after JWT generation — even if logging fails (best-effort), the auth response is still returned. A logging failure does not prevent login or token issuance.PASS
SEC-8Software IntegrityNo new external scripts or CDN resources added. All assets bundled via Vite.PASS
SEC-9Logging / MonitoringAudit log write failures are logged at WARN level via NestJS Logger — visible in server logs. The audit log itself IS the monitoring surface for this feature.PASS
SEC-10SSRFNo outbound HTTP calls triggered. Audit logging is purely DB-bound.N/A
SEC-11XSSReact escapes all rendered strings (entityTitle, user.fullName, metadata values). No dangerouslySetInnerHTML usage in ActivityLogPage. Status change metadata rendered via String() cast.N/A
