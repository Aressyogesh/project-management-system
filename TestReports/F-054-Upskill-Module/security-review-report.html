SEC-01 — Unauthenticated Static Serving of Evidence Files HIGH FIXED

backend/src/main.ts:23

Authorization Bypass / Data Exposure

10 / 10

express.static was mounted at /uploads before any NestJS guard could intercept the request, meaning every file under uploads/upskill/ (HR-sensitive certificates, DOCX files) was publicly downloadable by anyone knowing the UUID-based filename. Filenames are exposed in findAll / findOne API responses, making discovery trivial for authenticated users.

Authenticated user calls GET /api/v1/upskill/assignments, reads evidenceFilePath from the JSON response, then fetches GET /uploads/upskill/<uuid>.pdf with no JWT — bypassing the GET /upskill/assignments/:id/evidence auth guard entirely.

Added Express middleware before the static handler that intercepts /uploads/upskill and returns HTTP 403. Evidence files are exclusively served through the authenticated GET /upskill/assignments/:id/evidence endpoint.

SEC-02 — File Extension Spoofing via Client-Controlled originalname HIGH FIXED

backend/src/upskill/upskill.controller.ts:118

Insecure File Upload / Stored XSS Vector

9 / 10

The Multer fileFilter checked file.mimetype (client-controlled HTTP header) to validate file type, but used extname(file.originalname) (also client-controlled) to derive the stored filename extension. An attacker could submit a file with Content-Type: image/jpeg and filename: "shell.html" to store an HTML/XSS payload on the server with a .html extension. Combined with SEC-01, this file would be publicly accessible.

Replaced the extname(originalname) call with a server-side MIME-to-extension map (EVIDENCE_MIME_TYPES: Record<string, string>). The stored extension is now always one of .pdf, .docx, .png, or .jpg regardless of what the client sends in originalname.

SEC-03 — Manager IDOR: Any PM Can Approve / Reject Any Assignment Org-Wide HIGH FIXED

backend/src/upskill/upskill.service.ts:197–222

Insecure Direct Object Reference (IDOR) / Privilege Escalation

9 / 10

The approveAssignment and rejectAssignment service methods performed no check that the calling manager has organisational authority over the assignment. The only gate was the controller-level isManager() check, which returns true for any PROJECT_MANAGER on any project. A PM from Project A could approve assignments created for employees in Project B, fraudulently granting KPI score boosts (10/10 Growth & Innovation) to employees outside their remit.

PM on Project A guesses or enumerates a SUBMITTED assignment ID (UUIDs are hard to guess but can be obtained from findAll if PM knows the assignee ID). Sends PATCH /api/v1/upskill/assignments/<victim-id>/approve. Service approves with no scope check, bumping KPI for an unrelated employee.

Added creator-scope check to both approveAssignment and rejectAssignment: non-privileged callers (systemRole !== ADMIN | SUPER_USER) are required to be the createdById of the assignment. Admins and SuperAdmins can approve/reject org-wide. Two new unit tests (SEC-01, SEC-02) verify the ForbiddenException path.

SEC-04 — evidenceFilePath Exposed in List/Detail API Responses MEDIUM ACCEPTED — MITIGATED BY SEC-01 FIX

backend/src/upskill/upskill.service.ts:84–119, 122–141

Data Exposure / Information Leakage

8 / 10

The internal server filesystem path (uploads/upskill/<uuid>.pdf) is returned in findAll and findOne Prisma responses. With SEC-01 fixed (direct static access blocked), knowing the path no longer yields the file without a JWT. The field is retained in responses to support the frontend UpskillCard submission state indicator; removing it would require a schema projection change. Accepted as residual low risk post-SEC-01 fix — the path cannot be used to bypass auth.

SEC-05 — isCreator Permanently Grants Evidence Download Without State Check MEDIUM ACCEPTED — LOW RESIDUAL RISK

backend/src/upskill/upskill.service.ts:225–236

Authorization — Overly Broad Grant

8 / 10

The getEvidence method grants access via isCreator (assignment creator) permanently, with no check on assignment state or whether the creator still manages the employee. Accepted because: (1) the scope is bounded — only the specific PM who created the assignment gains access, not all PMs; (2) the manager reviewing evidence to make approve/reject decisions is the intended use case; (3) SEC-01 fix ensures only JWT-authenticated callers can reach the endpoint at all.
